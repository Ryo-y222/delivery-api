配車計画 (Dispatch Plan) 機能仕様書

ドキュメント 配車区間
ドキュメント / 配車計画

概要

配車計画 (Dispatch Plan) は、配送プラットフォームの中核機能であり、1台のトラック × 1日の運行を表現します。

各配車計画は複数の配車区間 (Trip Leg)を含み、複合輸送ネットワークにおける単一の物理的な運行単位として機能します。

主な特徴

  • トラック・ドライバーと日付の組み合わせで一意に識別
  • 複数の配車区間（往路・復路）を包含
  • ステータス管理による運行進捗追跡
  • 位置予測APIによるリアルタイム運行監視
  • 企業別・日付別の高速検索

データベーススキーマ

dispatch_plans テーブル

カラム名 型 制約 説明
id BIGINT PRIMARY KEY 配車計画ID
company_id BIGINT NOT NULL, FK 企業ID (companies.id)
vehicle_id BIGINT NOT NULL, FK 車両ID (vehicles.id)
driver_id BIGINT NOT NULL, FK ドライバーID (drivers.id)
plan_date DATE NOT NULL 運行予定日
status VARCHAR(20) NOT NULL, DEFAULT 'planned' ステータス (planned/in_progress/completed/cancelled)
note TEXT NULL 運行に関する注記・指示
created_at TIMESTAMP NOT NULL, DEFAULT CURRENT_TIMESTAMP 作成日時
updated_at TIMESTAMP NOT NULL, DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP 更新日時

インデックス

-- 高速検索用: 企業別・日付別の配車計画取得

リレーション

配車区間との関係

1つの配車計画は複数の配車区間を含みます (1:N 関係)。

  • 往路区間 (outbound): 荷物を積んだ状態で出発地から目的地へ
  • 復路区間 (return): 空車で目的地から出発地へ戻る、または別の荷物を探索

運行フロー図

1つの配車計画の典型的な構成:

配車計画ID=12345 (2026-03-20)
vehicle_id=3 (トラック③)
driver_id=101 (ドライバーA)
status=in_progress

┌─────────────────────────────────────────┐
│       配車計画 (Dispatch Plan)           │
│  トラック③ × 2026-03-20 × ドライバーA   │
└─────────────────────────────────────────┘
            │
    ┌───────┴───────┐
    ▼               ▼
┌─────────┐  ┌──────────┐
│往路区間  │  │復路区間   │
│(Leg 1)  │  │(Leg 2)   │
│outbound │  │return    │
└─────────┘  └──────────┘

Leg 1: 横浜 → 小田原
  leg_type: outbound
  cargo_status: loaded (荷物あり)
  departure_at: 09:00
  arrival_at: 10:30

Leg 2: 小田原 → 横浜
  leg_type: return
  cargo_status: empty_seeking (空車で次の荷物を探索)
  departure_at: 10:45
  arrival_at: 12:15

ステータス遷移

┌─────────┐
│ planned │  (計画済み)
└────┬────┘
     │
     ▼
┌─────────────┐
│ in_progress │  (運行中)
└────┬────────┘
     │
     ├──────────┐
     ▼          ▼
┌─────────┐  ┌──────────┐
│completed│  │cancelled │
└─────────┘  └──────────┘

遷移ルール:
- planned → in_progress: 運行開始時
- in_progress → completed: 全配車区間完了時
- planned / in_progress → cancelled: キャンセル時

データフロー

配車計画の作成から完了までのデータフロー:

1. 計画策定フェーズ
   計画者が企業・車両・ドライバーを選択
   → DispatchPlanを作成 (status: planned)

2. 配車区間登録フェーズ
   運行経路に基づいて複数のTripLegを登録
   → 往路: cargo_status=loaded
   → 復路: cargo_status=empty_seeking (マッチング対象)

3. 運行開始フェーズ
   → DispatchPlanのステータスをin_progressに更新
   → ドライバーがTripLegごとに位置情報を送信

4. マッチング実行フェーズ
   復路区間 (empty_seeking + public/company_only)
   がマッチングエンジンの対象になる
   → 荷主の配送依頼と照合

5. 運行完了フェーズ
   全TripLegがcompletedになると
   → DispatchPlanもcompletedに更新

設計ノート

indexed lookup performance

社企別・日付別インデックス idx_dispatch_plans_company_date により、特定企業の日次配車計画を高速に取得できます。典型的な検索パターン: "2026-03-20 のトラックの配車計画を全件取得"

配車区間との1:N関係

1つのDispatchPlanは複数のTripLegを保持します。これにより複雑な運行パターン（複数の積卸地点を持つ運行）を表現できます。

位置予測の用途

GET /predict エンドポイントは、ドライバーの現在位置と到着予測から遅延・早着を検出し、配送依頼のマッチング判定に利用されます。
