段階的情報開示機能 (Staged Information Disclosure)

概要

delivery-api の最重要セキュリティ & ビジネスロジックの一つ。マッチングの段階ごとに、会社情報をどの程度開示するかを制御します。検索段階では匿名名称のみ表示し、マッチング承認後にようやく実名・連絡先が見えるようになります。

主な特徴:

  • 4段階: 検索 → マッチング要求 → 承認 → 支払い完了
  • 各段階で異なる DTO（Data Transfer Object）を返す
  • 同一エンドポイントでもマッチング状況に応じて返却フィールドが変わる
  • 直接取引への動機付けを排除し、プラットフォーム内取引を促進
  • 実装: Service Layer で match status をチェック → 適切な DTO を選択

なぜ段階的開示が必要か

背景: フリーライディング問題

「運送会社の詳細まで表示すると、直接電話でやり取りされてしまう」

具体的な問題シナリオ

  1. ユーザーA が運送会社B を検索
  2. B の実名・電話・住所が即座に表示される
  3. A は「あ、直接電話した方が早い」と判断
  4. 直接やり取り → 決済なし → Stripe 手数料未回収
  5. レビューも記録されず、信頼スコアが蓄積されない
  6. プラットフォームの価値が毀損

対策思想

情報開示を段階的に行うことで、各ステップを通さないと詳細情報に辿り着けない仕組みにする。

  • 検索段階 → 匿名名称 + 評価スコア（信頼度の指標）
  • マッチング要求 → 「この企業と取引したい」という意思表示
  • 承認段階 → 実名を開示。但しチャットのみで連絡
  • 支払い完了 → 全情報開示（電話・住所等）。この時点で直接やり取りするインセンティブは薄れている

4つの段階と情報開示レベル

段階 マッチング状態 DTO 名 開示される情報
1. 検索 関係なし CompanySearchResponse display_name, rating_avg, review_count, plan
2. マッチング要求 match = pending CompanyMatchedResponse + total_deals（取引数）
3. 承認 match = approved CompanyApprovedResponse + name（実名）
4. 支払い完了 match = completed CompanyFullResponse + phone, address, email, website

DTO 構造体詳細

CompanySearchResponse（段階1: 検索時）

マッチング関係なく、検索時に返されるレスポンス。最小限の情報のみ。

type CompanySearchResponse struct {
  ID          uint      `json:"id"`
  DisplayName string    `json:"display_name"`     // 匿名名称
  RatingAvg   float32   `json:"rating_avg"`       // 平均評価
  ReviewCount uint      `json:"review_count"`     // レビュー数
  Plan        string    `json:"plan"`             // free/standard/premium
}

// 例
{
  "id": 50,
  "display_name": "配送太郎運送",
  "rating_avg": 4.67,
  "review_count": 15,
  "plan": "premium"
}

CompanyMatchedResponse（段階2: マッチング要求時）

マッチング pending 状態で、マッチング要求者に返す。total_deals（過去の取引数）を追加。

type CompanyMatchedResponse struct {
  ID          uint      `json:"id"`
  DisplayName string    `json:"display_name"`
  RatingAvg   float32   `json:"rating_avg"`
  ReviewCount uint      `json:"review_count"`
  Plan        string    `json:"plan"`
  TotalDeals  uint      `json:"total_deals"`      // 完了したマッチング数
}

// 例
{
  "id": 50,
  "display_name": "配送太郎運送",
  "rating_avg": 4.67,
  "review_count": 15,
  "plan": "premium",
  "total_deals": 42                              // 過去42件のマッチング実績
}

CompanyApprovedResponse（段階3: 承認時）

マッチング approved 状態で、当事者に返す。実名を開示。チャット準備段階。

type CompanyApprovedResponse struct {
  ID          uint      `json:"id"`
  DisplayName string    `json:"display_name"`
  Name        string    `json:"name"`             // 実名（正式法人名）
  RatingAvg   float32   `json:"rating_avg"`
  ReviewCount uint      `json:"review_count"`
  Plan        string    `json:"plan"`
  TotalDeals  uint      `json:"total_deals"`
}

// 例
{
  "id": 50,
  "display_name": "配送太郎運送",
  "name": "株式会社タイヨウ運送",
  "rating_avg": 4.67,
  "review_count": 15,
  "plan": "premium",
  "total_deals": 42
}

CompanyFullResponse（段階4: 支払い完了時）

マッチング completed 状態で、全ての情報を開示。直接連絡先を含む。

type CompanyFullResponse struct {
  ID          uint      `json:"id"`
  DisplayName string    `json:"display_name"`
  Name        string    `json:"name"`
  Phone       string    `json:"phone"`            // 電話番号

  Email       string    `json:"email"`            // メールアドレス
  Website     string    `json:"website"`          // ウェブサイト（オプション）

実装: Service Layer での DTO 切り替え

Go コード例

// CompanyService.GetCompanyInfo - マッチング状況に応じた DTO 選択
func (s *CompanyService) GetCompanyInfo(ctx context.Context, companyID, requesterID uint) (interface{}, error) {
  // 1. 対象会社を取得
  company, err := s.repo.GetCompany(ctx, companyID)
  if err != nil {
    return nil, err
  }

  // 2. リクエスターと対象会社の間にマッチングがあるか確認
  match, err := s.matchRepo.GetMatch(ctx, requesterID, companyID)
  if err != nil && err != gorm.ErrRecordNotFound {
    return nil, err
  }

  // 3. マッチング状態に応じて DTO を選択
  switch {
  case match == nil:
    // マッチング関係なし → CompanySearchResponse
    return &dto.CompanySearchResponse{
      ID:          company.ID,
      DisplayName: company.DisplayName,
      RatingAvg:   company.RatingAvg,
      ReviewCount: company.ReviewCount,
      Plan:        company.Plan,
    }, nil

  case match.Status == "pending":
    // マッチング要求状態 → CompanyMatchedResponse
    return &dto.CompanyMatchedResponse{
      ID:          company.ID,
      DisplayName: company.DisplayName,
      RatingAvg:   company.RatingAvg,
      ReviewCount: company.ReviewCount,
      Plan:        company.Plan,
      TotalDeals:  company.TotalDeals,
    }, nil

  case match.Status == "approved":
    // マッチング承認状態 → CompanyApprovedResponse
    return &dto.CompanyApprovedResponse{
      ID:          company.ID,
      DisplayName: company.DisplayName,
      Name:        company.Name,           // 実名開示
      RatingAvg:   company.RatingAvg,
      ReviewCount: company.ReviewCount,
      Plan:        company.Plan,
      TotalDeals:  company.TotalDeals,
    }, nil

  case match.Status == "completed" && match.PaymentStatus == "succeeded":
    // マッチング完了＆決済成功 → CompanyFullResponse（全情報）
    return &dto.CompanyFullResponse{
      ID:          company.ID,
      DisplayName: company.DisplayName,
      Name:        company.Name,
      Phone:       company.Phone,          // 電話開示
      Address:     company.Address,        // 住所開示
      Email:       company.Email,          // メール開示
      Website:     company.Website,        // ウェブサイト開示
      RatingAvg:   company.RatingAvg,
      ReviewCount: company.ReviewCount,
      Plan:        company.Plan,
      TotalDeals:  company.TotalDeals,
    }, nil

  default:
    // その他のステータス（rejected など）
    return &dto.CompanySearchResponse{...}, nil
  }
}

関連機能との連携

チャット機能（仕様書10）との連携

マッチング approved 状態でチャットルームが自動作成されます。ここで実名が見えるようになるため、チャット内での直接連絡先の交換をフィルタリングで防ぐ必要があります（→ 仕様書10: フィルタリング参照）。

評価・レビュー（仕様書11）との連携

マッチング完了後、review を投稿する際も CompanyFullResponse を用いられます。完了段階なので、会社の全情報を閲覧してからレビュー投稿が可能。

サブスクリプション（仕様書12）との連携

Premium プラン企業は検索結果での「優先表示」機能が利用可能。CompanySearchResponse の時点でも premium フラグが見えるため、ユーザーが高価値企業を識別できます。

フロントエンド実装上の注意

CORS & API キャッシュ

  • 同じエンドポイント（GET /companies/:id）でも、認証ユーザーのマッチング状態により返却内容が異なる
  • → API キャッシュはユーザー & 企業ID の複合キーで管理する必要がある
  • あるいは、キャッシュなしでリアルタイムフェッチ

UI/UX

  • 検索時: 匿名名称 + 評価スコアのみ表示。「詳細を見るにはマッチングリクエストを送信」というテキスト提示
  • マッチング要求送信後: 実名が段階的に表示される UI を工夫（例: 「リクエスト送信完了。相手の詳細情報が表示されました」という通知）
  • 完了後: 実名・電話・住所を完全表示。次回マッチングの際は検索段階からの再スタート

React 実装例

// CompanyCard.tsx
const CompanyCard: React.FC<{companyId: number}> = ({companyId}) => {
  const [company, setCompany] = useState(null);
  const {authUser} = useAuth();

セキュリティ & エッジケース

アクセス制御

  • マッチング当事者以外が /companies/:id にアクセスした場合、CompanySearchResponse のみ返す
  • マッチング状態を認証ユーザーID + 対象企業ID で検証（JWT トークンから user_id を抽出）

エッジケース

  1. rejected マッチング: 承認されなかったマッチングは、引き続き CompanySearchResponse レベル
  2. 複数マッチング: 同一ユーザー/企業間で複数 match が存在する場合、status が最も進んでいるものを使用
  3. マッチング削除後: マッチング削除（論理削除）後は、検索段階に戻る
  4. 企業情報更新:企業の phone / address が更新された場合、既存 match.completed マッチングでもリアルタイムで新情報が表示される

データ整合性

  • 企業の display_name と name は必ず異なること（バリデーション）
  • completed マッチングでも payment_status = "succeeded" に達していない場合は、CompanyApprovedResponse を返す（保守的）

デプロイ・運用上の考慮

ロールアウト

  • この機能はシステムの根幹であるため、全テーブル・エンドポイントで一括導入を推奨
  • 段階的ロールアウトの場合は、feature flag で制御（ON/OFF を分ける）

モニタリング

  • API レスポンスに含まれるフィールド数をモニタリング
  • CompanyFullResponse の返却頻度が異常に低い場合 → マッチング完了率の低下を示唆

バージョニング

  • API を /v1/ と /v2/ に分けている場合、両方で段階的情報開示を実装する
