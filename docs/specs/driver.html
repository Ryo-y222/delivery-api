ホーム / ドライバー管理仕様

概要

ドライバー管理モジュールは、配送プラットフォームにおけるドライバーの登録・管理・運用を担当します。 ユーザーシステムとは独立したドライバーテーブルを使用し、 免許情報・安全実績・資格情報を一元管理します。

本仕様では、ドライバーテーブル、日次点検テーブル、API エンドポイント、 および ユーザーシステムとの関係性について詳細に記載します。

データベーススキーマ

drivers テーブル

ドライバー固有の情報を管理するテーブルです。 users テーブルとは分離して管理され、 ドライバーに必要な特殊な属性（免許情報、資格、事故歴など）を保持します。

カラム名 データ型 制約 説明
id uint PRIMARY KEY ドライバーID（自動採番）
user_id uint NOT NULL
FOREIGN KEY		 ユーザーID（users テーブルとの関連）
company_id uint NOT NULL
FOREIGN KEY		 所属企業ID
license_number varchar(20) NOT NULL 運転免許証番号
license_class varchar(20) 免許区分（大型/中型/普通）
license_expiry date 免許有効期限
accident_count int 事故件数（累積）
violation_count int 違反件数（累積）
has_hazmat bool 危険物運送資格有無
has_forklift bool フォークリフト操作資格有無
created_at timestamp 作成日時
updated_at timestamp 更新日時

daily_inspections テーブル

ドライバーの日次点検記録を管理するテーブルです。 出勤時のアルコール検査・健康状態・整備状況などを記録します。

カラム名 データ型 制約 説明
id uint PRIMARY KEY 点検記録ID（自動採番）
driver_id uint NOT NULL
FOREIGN KEY		 ドライバーID
dispatch_plan_id uint NOT NULL
FOREIGN KEY		 配送計画ID
inspected_at timestamp 点検実施日時
alcohol_level float64 呼気中アルコール濃度（mg/100ml）
health_status varchar(20) 健康状態（good/poor）
note text 備考
inspector_id uint 点検者ID（ユーザーID）
created_at timestamp 作成日時

免許区分（license_class）の説明

区分 説明 運転可能車両
普通 普通自動車運転免許 軽トラ、一部の2tトラック
中型 中型自動車運転免許 2t、4t トラック
大型 大型自動車運転免許 すべてのトラック

健康状態（health_status）の説明

ステータス 説明 配送可能
good 健康。配送に従事可能な状態 〇 可能
poor 不良。体調不良などにより配送に従事不可 × 不可

関係性の説明

  • users ← 1:1 → drivers： 各ドライバーは1人のユーザーに対応します。 ユーザーはドライバーの認証・認可に使用されます。
  • companies ← 1:N → drivers： 1社は複数のドライバーを雇用できます。 各ドライバーは必ず1つの企業に所属します。
  • drivers ← 1:N → daily_inspections： 1ドライバーは複数の日次点検記録を持ちます。 毎日の点検が記録として蓄積されます。
  • dispatch_plans ← 1:N → daily_inspections： 1つの配送計画に対して1つの点検が関連付けられます。 点検はその日の配送計画と関連します。

ドライバーテーブルが users から分離されている理由

1. ドライバー固有のドメイン知識の分離

users テーブルはシステム全体の基本ユーザー情報を管理します。 一方、drivers テーブルはドライバーという特定の役割に関する情報を管理します。 この分離により、各エンティティの責務が明確になり、保守性が向上します。

2. ドライバー固有フィールドの管理

ドライバーには以下のような固有フィールドが必要です：

  • 運転免許情報（免許番号、区分、有効期限）
  • 安全実績（事故件数、違反件数）
  • 資格情報（危険物運送、フォークリフト操作）

これらのフィールドを users テーブルに含めると、 テーブルが肥大化し、ユーザー管理の複雑性が増します。

3. マルチロール対応

ユーザーが複数の役割を持つ可能性があります （例：管理者でありつつドライバー）。 分離することで、各ユーザーは複数のドライバー記録を持つ可能性を排除し、 データモデルを単純化します。

4. データベーススキーマの柔軟性

drivers テーブルを独立させることで、 ドライバー関連の機能追加や仕様変更が、 ユーザーシステム全体に影響を与えません。

5. アクセス制御と監査

ドライバー情報へのアクセスを細かく制御できます。 ドライバー関連の変更を独立して監査・ロギングすることが容易になります。

実装上の考慮事項

日次点検とドライバー出勤フロー

出勤時の流れは以下の通りです：

  1. ドライバーが出勤
  2. 管理者またはドライバー自身が日次点検を実施
  3. アルコール検査（呼気アルコール濃度測定）を実施
  4. 健康状態をチェック
  5. 点検記録を daily_inspections テーブルに記録
  6. 問題がなければ配送業務に従事
  7. 問題がある場合は配送を中止

免許有効期限の管理

license_expiry が過期している場合、 そのドライバーは配送業務に従事することはできません。 アプリケーションロジックで有効期限をチェックし、 期限切れドライバーを自動的に配送対象から除外する必要があります。

事故・違反件数の追跡

accident_count と violation_count は累積値です。 事故や違反が発生した場合、これらのカウントを増加させます。 これらの値が一定の閾値を超える場合、 ドライバーの配送業務を制限することが望ましいです。

アルコール検査値

alcohol_level は呼気中アルコール濃度（mg/100ml）です。 日本の法令では、0.15mg/100ml 以上は道路交通法違反となります。 0 に近いほど安全です。アプリケーション側で閾値を設定し、 一定値以上の場合は配送を禁止する必要があります。

資格情報の管理

has_hazmat と has_forklift は、 ドライバーの資格を示すフラグです。 配送計画作成時に、必要な資格を持つドライバーのみを配置する必要があります。
