チャット & フィルタリング機能 (Chat & Filtering)

概要

マッチング承認後、自動でチャットルームが作成されます。このチャット機能では、運送会社と荷主（または他の運送会社）が安全にコミュニケーションを行えます。プラットフォーム内でのやり取りを強制し、直接連絡先の交換を防ぐため、メッセージは正規表現に基づいてフィルタリングされます。

主な特徴:

  • マッチング承認時に自動でチャットルーム作成
  • チャットルーム = 1マッチング = 1ルーム（1対1対応）
  • メッセージの内容に基づくフィルタリング（電話番号、メールアドレス、URL）
  • フィルタリング違反の記録と追跡
  • メッセージ不変性（immutability）の実装
  • 正規表現パターンの動的管理

フィルタリングに引っかかったメッセージを記録。ユーザーの違反回数でスパム判定やアカウント停止を検討。

メッセージフロー

ユーザーがメッセージ送信
         ↓
[フィルタリング処理]
① メッセージ内容を取得
② blocked_patterns テーブルから全アクティブパターンを取得
③ 各パターンで正規表現マッチを試行
         ↓
マッチあり？
├─ YES → status = "filtered"
│         create user_violations レコード
│         クライアントに警告メッセージを返す
│         （※ メッセージは保存されるが、相手には表示されない）
│
└─ NO  → status = "sent"
         メッセージ保存完了
         相手にメッセージ配信

フィルタリングの目的

問題の背景

「会社情報を出せないよう検知」 —

プラットフォームを通さずに、ユーザー同士が直接連絡を取ってしまう行為を防ぎたい。

ビジネス上の課題

  • マネタイズの喪失: 直接取引されると、Stripe 決済を経由されず、プラットフォーム手数料が失われる
  • 品質管理の低下: 直接取引ではレビュー・評価が記録されず、信頼スコアが蓄積されない
  • 税務・法的リスク: 無追跡取引による脱税の温床になる可能性
  • サポート不在: トラブル時にプラットフォーム側がサポートできない

対策の考え方

メッセージレベルでの防止ではなく、段階的情報開示と組み合わせることで、直接取引の動機自体を減らす戦略も採用しています（→ 仕様書13を参照）。

チャットルーム自動作成

トリガー

マッチングのステータスが pending から approved に変わったとき。

処理フロー

  1. マッチング承認APIが呼ばれる（PATCH /api/v1/matches/:id）
  2. ステータスを approved に更新
  3. トランザクション内で chat_rooms テーブルにレコード作成
  4. 初期状態: status = "active"
  5. 両者にWebSocketまたはポーリングで「チャットルーム作成」通知を送信

留意事項

  • マッチング承認をロールバックした場合、チャットルームも削除（段階的削除）
  • マッチング完了後、チャットルームは closed になる（重要: メッセージ送信を禁止する仕様）

メッセージの不変性（Immutability）

設計原則

一度送信されたメッセージは、更新されない・削除されない。

利点

  • 監査可能性: 違反メッセージの歴史が完全に追跡可能
  • データ整合性: メッセージIDの参照が壊れない
  • 法務対応: チャット履歴がエビデンスとして使える

実装

  • chat_messages テーブルに updated_at カラムなし
  • APIレベルでも UPDATE / DELETE エンドポイント不提供
  • アプリケーション層でバリデーション: 送信済みメッセージへのアクセス試行を拒否

削除が必要な場合

ユーザーが削除リクエストを出した場合は、以下の対応を取る:

  1. メッセージ内容をクリア: "[削除済み]" に置き換える
  2. 新しい履歴レコード（audit_log）を作成
  3. 実際のレコードは論理削除フラグで保持（または定期的アーカイブ）
