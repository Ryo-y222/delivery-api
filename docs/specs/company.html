1. 概要

企業管理機能は、配送プラットフォームに登録された配送企業（配送業者）の情報を一元管理します。 このシステムの主要な特徴は、段階的な情報開示により、不正利用やシステム悪用を防止することです。

情報開示の2段階制度

なぜ DisplayName と Name を分けるのか

セキュリティ上の理由：

  • 直接接触の防止： 検索段階では企業の匿名名（DisplayName）のみを表示し、本当の企業名（Name）は隠す
  • システムタダ乗りの防止： マッチング承認前に実名を明かすと、ユーザーがシステムを経由せず直接連絡する可能性がある
  • 支払いベースの開示： 支払い完了後のみ、電話番号やメールアドレスなどの詳細情報を提供

開示ステージの流れ

  • 段階1（検索）： DisplayName（匿名名）のみ表示
  • 段階2（承認）： マッチング承認後に Name（実名）を表示
  • 段階3（支払い）： 支払い完了後に Phone、Address、Email を表示

2. データベーススキーマ

companies テーブルは、配送企業の全ての情報を保持します。

テーブル構造

カラム名 型 制約 説明
id UUID / BIGINT PRIMARY KEY 企業の一意なID。システム内部で使用
name VARCHAR(255) NOT NULL, UNIQUE 企業の実名。支払い後のみ表示。ユーザーテーブルの company_id で参照
display_name VARCHAR(255) NOT NULL 企業の匿名名（表示名）。検索結果に使用。直接連絡を防ぐ
address TEXT NULLABLE 企業の住所。支払い後のみ表示
headquarters_lat DECIMAL(10, 8) NULLABLE 本社所在地の緯度。配送計画や地理情報クエリに使用
headquarters_lng DECIMAL(11, 8) NULLABLE 本社所在地の経度。配送計画や地理情報クエリに使用
phone VARCHAR(20) NULLABLE 企業の電話番号。支払い後のみ表示。フォーマット: +81-XX-XXXX-XXXX
email VARCHAR(255) NULLABLE 企業のメールアドレス。支払い後のみ表示
logo_url VARCHAR(500) NULLABLE 企業ロゴの URL。検索結果に表示可能
rating_avg DECIMAL(2, 1) DEFAULT 0 平均レーティング。1.0 ～ 5.0 の範囲。検索結果のソート機準
total_deals INT DEFAULT 0 完了した配送案件の総数。信用スコアの指標
plan ENUM('free', 'standard', 'premium') DEFAULT 'free' 企業のサブスクリプションプラン。機能制限と API クォーターの根拠
created_at TIMESTAMP NOT NULL, DEFAULT CURRENT_TIMESTAMP 企業レコードの作成日時
updated_at TIMESTAMP NOT NULL, DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP 企業レコードの最終更新日時

インデックス

-- 企業検索の高速化
CREATE INDEX idx_display_name ON companies(display_name);

-- レーティングでのソート
CREATE INDEX idx_rating_avg ON companies(rating_avg DESC);

-- 実名検索（管理画面）
CREATE INDEX idx_name ON companies(name);

-- ユーザーテーブルでの参照整合性
CREATE INDEX idx_name_for_fk ON companies(name);

-- 地理情報クエリ
CREATE SPATIAL INDEX idx_headquarters ON companies(headquarters_lat, headquarters_lng);

ユーザーが企業情報を取得するにあたり、段階的に開示される情報が異なります。 以下の表は、各ステージでどの情報が見えるかを示します。

display_name ✓ 表示 ✓ 表示 ✓ 表示
logo_url ✓ 表示 ✓ 表示 ✓ 表示
rating_avg ✓ 表示 ✓ 表示 ✓ 表示
total_deals ✓ 表示 ✓ 表示 ✓ 表示

実装指針

API レスポンスのフィルタリング

各エンドポイントの実装では、リクエストユーザーのマッチング承認ステータスと支払いステータスに基づいて、 レスポンスから該当フィールドを除外する必要があります。 

// 疑似コード
function filterCompanyResponse(company, userStatus) {
    const result = {
        id: company.id,
        display_name: company.display_name,
        logo_url: company.logo_url,
        rating_avg: company.rating_avg,
        total_deals: company.total_deals
    };

    if (userStatus.isMatchingApproved) {
        result.name = company.name;
    }

    if (userStatus.isPaymentComplete) {
        result.phone = company.phone;
        result.email = company.email;
        result.address = company.address;
        result.headquarters_lat = company.headquarters_lat;
        result.headquarters_lng = company.headquarters_lng;
    }

    return result;
}

4. API エンドポイント

エンドポイント一覧

  • POST /api/v1/companies - 企業を作成
  • GET /api/v1/companies/:id - 企業情報を取得
  • GET /api/v1/companies/search - 企業を検索
  • PUT /api/v1/companies/:id - 企業情報を更新

1. 企業を作成

POST /api/v1/companies

新しい企業をシステムに登録します。企業の実名と匿名名の両方を設定します。

リクエスト

パラメータ 型 制約 説明
name string 必須 企業の実名（一意）。1～255文字
display_name string 必須 企業の匿名表示名。1～255文字。複数の企業が同じ display_name を持つ可能
address string オプション 企業の住所
headquarters_lat number オプション 本社の緯度（-90.0 ～ 90.0）
headquarters_lng number オプション 本社の経度（-180.0 ～ 180.0）
phone string オプション 企業の電話番号。フォーマット: +81-XX-XXXX-XXXX
email string オプション 企業のメールアドレス
logo_url string オプション 企業ロゴの URL（有効な URL 形式）
plan enum オプション プラン: free / standard / premium。デフォルト: free

3. 企業を検索

GET /api/v1/companies/search?q=&sort=&limit=

display_name で企業を検索します。複数の企業がヒットする場合、rating_avg でソートされます。 このエンドポイントでは常に段階1の情報（display_name のみ）が返されます。

クエリパラメータ

パラメータ 型 デフォルト 説明
q string - 必須 検索キーワード
sort enum rating ソート順: rating / deals / recent
limit integer 20 返す結果数（1～100）

6. ビジネスルール

ルール1: 企業名の一意性

name カラムは UNIQUE 制約を持ちます。 システム内でどの企業も同じ実名を持つことはできません。 これにより、ユーザーテーブルで正確な company_id 参照が可能になります。

ルール2: display_name の重複許容

display_name は複数の企業が同じ値を持つことができます。 これは検索結果で複数の企業が同じ匿名名で表示される可能性があることを意味します。 ユーザーは rating_avg や total_deals を基準に選択します。

ルール3: デフォルトプラン

企業の plan カラムのデフォルト値は 'free' です。 新規登録企業は自動的に無料プランに割り当てられ、 企業自身がアップグレードを申請することで standard または premium に変更できます。

ルール4: 地理情報の許容性

headquarters_lat と headquarters_lng は NULL 可能です。 国際的な企業や住所がない企業も登録できるよう、地理情報は必須ではありません。 ただし、配送計画での地理情報クエリを利用する企業は、これらの値の設定を推奨されます。

ルール5: rating_avg の初期値

新規登録企業の rating_avg は 0 です。 企業が初めての配送を完了し、レビュー評価を受けることで、 rating_avg が更新されます。0 の企業は検索結果のソートで下位に表示されます。

ルール6: total_deals のインクリメント

total_deals は配送が完了するたびに 1 ずつ増加します。 この値は企業の信用スコアの一部として機能し、検索フィルタリングや ユーザーの選択判断に影響します。

ルール7: 所有権と更新権限

企業情報は以下の条件でのみ更新可能です：

  • 企業自身の認証されたユーザー（company_id が一致）
  • プラットフォーム管理者（admin ロール）

name カラムは企業作成後は更新不可です。 これは参照整合性を保証するため、ユーザーテーブルが company.name を参照しているためです。

ルール8: 情報開示の厳格性

情報開示ステージは API レスポンスで厳格に適用されなければなりません。 クライアント側で情報を隠すのではなく、サーバー側で確実にフィルタリングする必要があります。 これにより、API を直接呼び出すクライアントであっても、段階的な開示が保証されます。

ルール9: logo_url のバリデーション

logo_url は有効な HTTPS URL である必要があります。 HTTP URL は受け入れられません。これはセキュリティ上の理由（中間者攻撃の防止）と、 ブラウザの Mixed Content ポリシーへの準拠です。

ルール10: 電話番号のフォーマット

phone カラムの推奨フォーマット：

  • 国際フォーマット: +81-3-1234-5678（日本の場合）
  • データベースには正規化されたフォーマットで保存
  • クライアント側で地域に応じた表示フォーマットに変換
