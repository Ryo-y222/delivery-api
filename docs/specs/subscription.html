サブスクリプション機能 (Subscription & Billing)

概要

delivery-api のマネタイズの核となるサブスクリプション機能です。Stripe Billing を連携し、運送会社は3つのプランの中から選択できます。free プランは初期ユーザー向けの制限機能、standard/premium は月額課金で機能解放。

主な特徴:

  • 3プラン: free / standard / premium
  • Stripe Billing との完全統合
  • 月額自動課金（月1回）
  • free プランの月間マッチング数制限（3件/月）
  • 支払い失敗時の past_due 状態
  • Webhook による自動更新

各プランの詳細

Free プラン

  • 対象: スタートアップ企業、試験導入企業
  • 月額: ¥0
  • マッチング制限: 月3件（超過するとマッチングリクエストを受け付けない）
  • 対応モード: ソロモードのみ
  • 機能制限: チャット、決済、API は利用不可
  • 使用例: 自社の配車計画（dispatch_plan）を作成して、配車ボードを使う。マルチモードへのアップグレードを促す

Standard プラン

  • 対象: 中小運送会社、月単位で安定した案件がある企業
  • 月額: ¥9,800
  • マッチング: 無制限
  • 対応モード: ソロ + マルチ（マッチング市場へのアクセス）
  • 機能: チャット、決済、レビュー等の全基本機能
  • 優先度: 通常表示（premium 企業よりも下）

Premium プラン

  • 対象: 大規模運送会社、API 連携が必要な企業
  • 月額: ¥29,800
  • マッチング: 無制限
  • 対応モード: ソロ + マルチ + API
  • 機能:
    • 全基本機能
    • REST API による自動連携（配車計画の自動作成等）
    • マッチング検索時の「優先表示」（検索結果の上位に表示）
    • 高度な analytics & レポート
    • 優先カスタマーサポート

Free プランのマッチング数制限

制限の仕組み

current_period_start から current_period_end の期間内に、そのユーザーが作成したマッチング数（status = pending/approved 以上）をカウント。

チェック箇所

マッチングリクエスト作成時（POST /api/v1/matches）に以下をバリデーション:

  1. subscription.plan = "free" か確認
  2. この月のマッチング数 ≥ 3 か確認
  3. YES → 422 Unprocessable Entity 返却、マッチングリクエスト作成不可
  4. NO → マッチング作成処理を続行

SQL 例

SELECT COUNT(*) as match_count
FROM matches
WHERE requester_company_id = ?
  AND created_at >= ?  -- current_period_start
  AND created_at < ?   -- current_period_end
  AND status != 'rejected'

月初め処理

Stripe Webhook customer.subscription.updated で current_period_start / current_period_end が更新される。マッチング制限カウンターは自動的にリセット。

Stripe Billing 連携フロー

1. ユーザーが「プラン選択」画面で plan を選択
         ↓
2. [Free 選択の場合]
   - subscription.plan = "free"
   - stripe_subscription_id = NULL
   - status = "active"
   - そのまま完了
         ↓
3. [Standard/Premium 選択の場合]
   - Stripe Checkout Session 作成
   - client_reference_id = company_id (追跡用)
   - success_url = https://app/subscription/success
   - cancel_url = https://app/subscription
         ↓
4. クライアントを Stripe Hosted Checkout へリダイレクト
         ↓
5. ユーザーがカード情報を入力 → 確認
         ↓
6. Stripe がカード処理実行
         ↓
7. Webhook: checkout.session.completed
   - Stripe Customer / Subscription 作成
   - DB に stripe_subscription_id を保存
         ↓
8. Webhook: invoice.paid
   - subscription.status = "active"
   - subscription.current_period_start/end を更新
   - DB コミット
         ↓
9. クライアント側で /subscription/success にリダイレクト
   - ユーザーに「アップグレード完了」を表示

API エンドポイント

POST /api/v1/subscriptions/checkout
Stripe Checkout Session を作成（standard/premium プラン）

リクエスト例

{
  "plan": "standard"
}

レスポンス例

{
  "session_id": "cs_test_a1234567890123456789",
  "checkout_url": "https://checkout.stripe.com/pay/cs_test_a1234567890123456789"
}

ステータスコード

  • 200 OK - Checkout Session 作成成功
  • 400 Bad Request - plan パラメータなし
  • 422 Unprocessable Entity - 既に有料プランを利用中
GET /api/v1/subscriptions/current
現在のサブスクリプション情報を取得

レスポンス例

{
  "id": 100,
  "company_id": 50,
  "plan": "standard",
  "status": "active",
  "current_period_start": "2026-03-01T00:00:00Z",
  "current_period_end": "2026-03-31T23:59:59Z",
  "created_at": "2026-02-15T10:00:00Z",
  "updated_at": "2026-03-01T00:00:00Z",
  "billing_portal_url": "https://billing.stripe.com/..."
}
POST /api/v1/subscriptions/cancel
サブスクリプションをキャンセル（free プランへダウングレード）

リクエスト例

{
  "reason": "ビジネス休止中"  // オプション（カスタマーサポート用）
}

レスポンス例

{
  "id": 100,
  "plan": "free",
  "status": "cancelled",
  "cancelled_at": "2026-03-20T12:00:00Z"
}
POST /api/v1/webhooks/stripe
Stripe Webhook エンドポイント（invoice.paid, customer.subscription.updated）

処理対象イベント

  • invoice.paid - 請求成功時。subscription.status = "active"
  • customer.subscription.updated - サブスク更新（period 変更等）
  • customer.subscription.deleted - キャンセル時。subscription.status = "cancelled"
  • invoice.payment_failed - 支払い失敗時。subscription.status = "past_due"

Webhook シグネチャ検証

// Go 例
sig := r.Header.Get("Stripe-Signature")
event, err := webhook.ConstructEvent(body, sig, endpointSecret)
// 検証失敗時は 400 Bad Request

[Free プラン]

   │
   └─→ [Checkout Session 作成]
          │
          ├─→ [キャンセル] → [Free プランへ戻る]
          │
          └─→ [カード処理] → [invoice.paid Webhook]
                        │
                        └─→ [Active]
                             │
                             ├─→ [支払い失敗] → [past_due]
                             │                   │
                             │                   └─→ [再試行成功] → [Active]
                             │
                             └─→ [ユーザーキャンセル] → [Cancelled]

past_due 状態の処理

発生条件

毎月の自動課金時にカード決済に失敗した場合。

期間

Stripe のデフォルト: 最大 3 回自動リトライ。その後 30 日間 past_due 状態が続く。

ユーザーへの影響

  • マッチング機能: マッチングリクエスト送信不可。既存マッチングの完了までは利用可
  • UI 通知: ダッシュボードに「支払い方法を更新してください」というバナーを表示
  • 支払い方法更新: Stripe Billing Portal から再入力可能

API 側の処理

マッチングリクエスト作成時に以下をチェック:

subscription.status == "past_due" → 422 Unprocessable Entity
{
  "error": "支払い情報が不足しています。Billing Portal で更新してください。",
  "billing_portal_url": "https://billing.stripe.com/..."
}

ダウングレード時の動作

Standard/Premium → Free へのダウングレード

  1. ユーザーが /subscriptions/cancel を呼び出し
  2. Stripe Subscription をキャンセル（現在の period_end で終了）
  3. DB に Webhook が返ってきた時点で:
    • subscription.plan = "free"
    • subscription.status = "cancelled"
    • subscription.stripe_subscription_id = NULL
    • subscription.current_period_start = 今月初日
    • subscription.current_period_end = 来月初日 - 1 日
  4. 次の月初めから月3件制限が適用される

注意点

Stripe の仕様上、キャンセル後も current_period_end まで access が続く場合がある。その期間は standard/premium 機能にアクセス可能。
