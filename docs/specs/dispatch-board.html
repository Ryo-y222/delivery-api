📍 配送ボード & 予測位置フィーチャー

仕様書 / API / 配送ボード
🎯 プラットフォームの最大の差別化要因

GPS なし で配送トラックの位置を予測します。
出発時刻・到着予定時刻 + ルートデータから、現在位置を算出・補間します。

1. 概要

配送ボードは、複数のトラック配送ルートを 1 つのマップ上にリアルタイムで表示し、 各トラックの現在位置を 予測 するシステムです。 GPS は使用せず、出発時刻・到着予定時刻・ルートデータから位置を算出します。

なぜ GPS ではなく予測を使うのか

  • ドライバーが嫌がる： GPS 24 時間監視は信頼を損ないます
  • ドライバー不足： 監視を強制できない現状があります
  • 十分な精度： 社内配送管理には、ざっくりした位置予測で十分です
  • プライバシー： 個人のプライバシーを尊重できます
  • 低コスト： GPS デバイス購入・維持費不要です
💡 ワンタップ報告で精度向上
出発・休憩開始/終了・到着を、ドライバーが任意で報告できます。 実際の発生時刻が記録され、予測の精度が大幅に向上します。

2. 予測位置アルゴリズム

ルート上の現在位置を以下の手順で計算します：

ステップバイステップ説明

  1. ルート取得： Google Routes API の computeRoutes で 出発地 → 目的地のルート を取得
  2. ポリラインデコード： ルートの polyline を座標の列に変換
  3. ステップ別の経過時間を累積： 各ステップの duration を積み上げて、 ルート上の各地点での累積時間を記録
  4. 経過時間を計算： 出発時刻 から 現在時刻 までの差分を算出
  5. ルート上の位置を特定： 累積時間がちょうど経過時間に一致する ポリラインのセグメントを見つける
  6. 補間： そのセグメント内で線形補間し、正確な座標 (lat, lng) を求める

2つの時間計算アプローチ

アプローチ 説明 推奨
A) Google 予測ベース Google Routes API の traffic-aware duration を使用。 実際の渋滞を考慮した移動時間を使う。 ✅ 推奨
B) スケジュール固定ベース 出発時刻と到着時刻から逆算。 ルート進行度が時間進行度と同じと仮定 (50% 時間 = 50% 進行度) シンプル
⚠️ トラフィック課金回避
Google Routes API の traffic-aware option は課金対象です。 必要に応じて、スケジュール固定ベースで対応できるよう設計しています。

マップ上の要素

  • ポリライン： ルートを色分け (赤/青/緑など) で表示
  • マーカー： 出発地 (◆)、現在位置 (📍)、到着地 (◆)
  • トラックアイコン： 🚚 で各トラックの予測位置を表示

トラック一覧テーブル

  • ID： トラック ID
  • ナンバープレート： トラック識別
  • ルート： 割り当てられたルート (A/B/C...)
  • 進捗度： ルート全体に対する進行度 (0～100%)
  • 状態： 移動中 / 到着予定 / 到着済み など
  • 帰り荷： 返却荷物の有無 (無/あり/確認中)
  • アクション： 詳細表示ボタン

4. Trackings テーブル

ワンタップ報告や自動記録をすべて記録するテーブル。 この履歴から、予測位置の精度を継続的に向上させます。

カラム 型 説明 備考
id UUID 一意の記録ID 主キー
dispatch_plan_id UUID 配送計画ID 外部キー
trip_leg_id uint (nullable) 配送ルートのレッグID 出発/到着など特定イベント用。通常NULL
lat decimal(10,7) 緯度 ワンタップ報告時は出発地/到着地の座標
lng decimal(10,7) 経度 ワンタップ報告時は出発地/到着地の座標
recorded_at timestamp 実際のイベント発生時刻 ドライバーが報告した時刻 (重要！)
event_type enum イベント種別 departure, rest_start, rest_end, arrival, manual_report
created_at timestamp DB記録時刻 自動タイムスタンプ
💾 SQL スキーマ例 
CREATE TABLE trackings (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    dispatch_plan_id UUID NOT NULL REFERENCES dispatch_plans(id),
    trip_leg_id UINT,
    lat DECIMAL(10, 7) NOT NULL,
    lng DECIMAL(10, 7) NOT NULL,
    recorded_at TIMESTAMP NOT NULL,
    event_type VARCHAR(20) NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (dispatch_plan_id) REFERENCES dispatch_plans(id)
);

GET /api/v1/dispatch-plans/:id/predict

5.2 ワンタップ報告の登録

POST /api/v1/trackings

ドライバーが出発・休憩・到着を報告します。

リクエスト

{
  "dispatch_plan_id": "uuid-xxx",
  "trip_leg_id": 1,
  "event_type": "departure",  // departure, rest_start, rest_end, arrival
  "lat": 35.6895,
  "lng": 139.6917,
  "recorded_at": "2026-03-20T09:00:15Z"
}

レスポンス (成功)

{
  "status": "success",
  "tracking_id": "uuid-tracking-xxx",
  "message": "出発を記録しました",
  "recorded_at": "2026-03-20T09:00:15Z",
  "event_type": "departure"
}

レスポンス (エラー)

{
  "status": "error",
  "code": "INVALID_EVENT_TYPE",
  "message": "event_type は departure, rest_start, rest_end, arrival のいずれかです"
}
🔄 自動再計算
この POST 後、サーバーは自動的に dispatch plan の予測を再計算します。 API クライアントは 60 秒以内に GET predict を再呼び出して、更新された位置を取得できます。

6. ワンタップ報告フロー

ドライバーが 1 タップで現在のイベント (出発・休憩開始・終了・到着) を報告し、 実際の発生時刻が記録されます。これにより、予測精度が大幅に向上します。

報告可能なイベント

  • 出発 (departure)： 配送拠点から出発
  • 休憩開始 (rest_start)： トラックが停止・休憩
  • 休憩終了 (rest_end)： 休憩から再開
  • 到着 (arrival)： 目的地に到着

ポリラインのデコード

Google Polyline Algorithm Format を使って、 圧縮された polyline 文字列を座標の列に変換します。

デコード例 (疑似コード)

function decodePolyline(encoded) {
    let points = [];
    let lat = 0, lng = 0;

    for (let i = 0; i < encoded.length; i++) {
        let shift = 0, result = 0;
        let char;

        do {
            char = encoded.charCodeAt(i++) - 63;
            result |= (char & 0x1f) << shift;
            shift += 5;
        } while (char >= 0x20);

        let dlat = (result & 1) ? ~(result >> 1) : (result >> 1);
        lat += dlat;

        shift = 0;
        result = 0;

        do {
            char = encoded.charCodeAt(i++) - 63;
            result |= (char & 0x1f) << shift;
            shift += 5;
        } while (char >= 0x20);

        let dlng = (result & 1) ? ~(result >> 1) : (result >> 1);
        lng += dlng;

        points.push({
            lat: lat / 1e5,
            lng: lng / 1e5
        });
    }

    return points;
}

8. 精度向上のコツ

GPS なし予測の精度を向上させるための施策：

🎯 ワンタップ報告の活用
出発・到着を実際に報告することで、予測の起点と終点が実データになります。 最も効果的な精度向上手段です。
📍 チェックポイント報告
ルート中盤で「現在ここ」と報告。複数の中継地点データで精度向上。
⏱️ 遅延手動入力
渋滞や待機時間を手動で入力。以降の予測が調整されます。
📊 履歴分析
過去の trackings データから、ドライバーごと・ルートごとの 平均遅延を学習。予測時間を調整。

推奨実装フロー

  1. 配送開始時に、ドライバーが [出発] をタップ → 実際の departure_time が記録
  2. 配送中、自動的に 60 秒ごとに GET predict を実行
  3. 重要な地点 (高速道路出入口など) で [チェックポイント] タップ
  4. 到着時に [到着] をタップ
  5. サーバーが trackings 履歴を分析し、次回の予測をさらに精密化

9. 制限事項・既知の問題

⚠️ 重要 以下のシナリオでは予測精度が大幅に低下します：

ルート途中での問題

  • 大きな渋滞： 予測では平均速度を想定するが、実際の渋滞は予測不可。 → 対策：ドライバーが遅延を手動報告
  • 予期しない停止： 故障・事故・突然の業務追加など。 → 対策：ワンタップで「その他」イベント報告
  • ルート変更： 出発後に別ルートに変更した場合、予測は古いルートベース。 → 対策：新ルートで API を再呼び出し

天候・リアルタイム要因

  • 天候悪化による速度低下は考慮できません
  • 道路工事・交通規制は事前登録が必要

改善案

  • 機械学習を用いた過去データの分析
  • リアルタイムの交通情報 API との連携 (追加課金あり)
  • ドライバー間のデータ共有と学習

10. まとめ

✨ 配送ボードの価値
  • GPS 非依存で、ドライバーのプライバシーを尊重
  • 低コスト (デバイス不要、API 課金最小化)
  • ワンタップ報告で実用的な精度を実現
  • 配送業界の人手不足に対応 (強制監視不要)
