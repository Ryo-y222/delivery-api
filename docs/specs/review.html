レビュー・評価機能 (Review & Rating)

概要

マッチングが完了した後、両者がお互いの相手を評価できる仕組みです。重要な点として、評価の対象は「人」ではなく「会社」です。ドライバーは日々変わりますが、会社の信頼スコアは累積されます。

主な特徴:

  • 1人（または会社）が1つのマッチングに対して1回だけレビュー可能
  • 評価対象は「会社」（運送会社 or 荷主企業）
  • 星評価（1〜5）とテキストコメント
  • マッチング完了後のみレビュー可能
  • companies テーブルの rating_avg を自動更新（キャッシュ / デノーマライズ）
  • レビュー履歴は削除不可（監査対応）

なぜ「人」ではなく「会社」か

背景

運送業界では、ドライバーが頻繁に変わります。毎日異なるドライバーが配送に当たるため、個人の評価では意味がありません。

評価スコアの使い方

  • マッチング検索時の表示: 「ABC運送 ★4.5 (98件）」と表示。ユーザーはスコアの高い会社を優先選択
  • ソート・フィルタ: 検索結果を「評価の高い順」にソート可能
  • アルゴリズム側: premium プラン会社の「優先表示」にも評価スコアを組み込み
  • 新規会社の課題: 0件の新規会社は不利。スタートアップ支援として initial_boost スコアを付与することも検討

ドライバーはどうする？

別フェーズで実装予定。現在は会社単位の評価に集中。

データベーススキーマ

reviews テーブル

カラム名 型 説明 制約
id uint (PK) レビューID PRIMARY KEY, AUTO_INCREMENT
match_id uint (FK) マッチングID FOREIGN KEY to matches
reviewer_id uint (FK) レビュアー（送信者）のユーザーID FOREIGN KEY to users
reviewee_company_id uint (FK) 被評価会社ID FOREIGN KEY to companies
rating tinyint (1-5) 星評価 1 / 2 / 3 / 4 / 5
comment text コメント / フィードバック 最大 500 文字（オプション）
created_at timestamp 作成日時 DEFAULT CURRENT_TIMESTAMP

ユニーク制約

複合ユニークインデックス: UNIQUE(match_id, reviewer_id)

⇒ 1人が1つのマッチングに対して複数回レビューできない

companies テーブル（抜粋）

reviews テーブルと連動する、既存の companies テーブルに追加される評価スコア関連カラム。

カラム名 型 説明 備考
rating_avg decimal(3,2) 平均評価スコア 1.00 〜 5.00。取得時に動的計算 or キャッシュ
review_count uint レビュー総数 キャッシュ用。reviews テーブルの COUNT(id) と同期
rating_updated_at timestamp 評価スコア更新日時 新規レビュー追加時に更新

評価分布の追跡

rating_distribution テーブル（オプション）

評価の分布を高速取得するため、キャッシュテーブルを検討。

カラム名 型 説明
id uint (PK) ID
company_id uint (FK) 会社ID
rating tinyint (1-5) 評価（★）
count uint この評価の件数
updated_at timestamp 更新日時

使用例: フロントエンドで「★5が45件、★4が50件...」と表示するための快速参照用。

レビュー可能条件

必須条件

  1. マッチングのステータスが completed に遷移
  2. 決済が成功（payment status = succeeded）
  3. レビュアーがマッチングに参加していた当事者（reviewer_id = match.requester_id or match.receiver_id）
  4. 同じ reviewer_id + match_id の組み合わせでまだレビューが存在しない

タイムスパン

マッチング完了後、30日以内にレビュー投稿可能。それ以降は POST エンドポイントで 410 Gone を返す。

API エンドポイント

POST /api/v1/reviews
新規レビューを投稿

リクエスト例

{
  "match_id": 1000,
  "rating": 5,
  "comment": "丁寧な配送対応でした。次回もお願いしたいです！"
}

レスポンス例（201 Created）

{
  "id": 5001,
  "match_id": 1000,
  "reviewer_id": 10,
  "reviewee_company_id": 50,
  "rating": 5,
  "comment": "丁寧な配送対応でした。次回もお願いしたいです！",
  "created_at": "2026-03-20T15:30:00Z"
}

エラーレスポンス例

// 409 Conflict - 既にレビュー済み
{
  "error": "既にこのマッチングに対するレビューを投稿しています"
}

// 410 Gone - 30日を超過
{
  "error": "このマッチングのレビュー投稿期限（30日）が過ぎています"
}

// 422 Unprocessable Entity - マッチングが完了していない
{
  "error": "マッチングがまだ完了していません。completed ステータス後にレビューを投稿してください"
}

クエリパラメータ

評価スコアの計算と更新

計算式

平均評価スコア（rating_avg）= SUM(rating) / COUNT(reviews)

例: (5 + 5 + 4 + 4 + 4) / 5 = 4.4

更新タイミング

  1. 新規レビューが POST されたとき
  2. トランザクション内で:
    • reviews テーブルに INSERT
    • rating_distribution を UPDATE
    • companies.rating_avg と companies.review_count を UPDATE
    • companies.rating_updated_at を現在時刻に UPDATE
  3. 全てコミット、またはロールバック

パフォーマンス最適化

  • キャッシュ戦略: Redis に company:{id}:rating_avg をキャッシュ。TTL = 1時間
  • バッチ更新: リアルタイム計算ではなく、1時間ごとにバッチで再計算も検討
  • 分散トランザクション: companies テーブル更新が遅い場合、非同期 Worker で更新

レビュー投稿フロー

マッチング完了（status = completed, payment = succeeded）
         ↓
ユーザーAが「レビュー投稿」ボタンをクリック
         ↓
[バリデーション]
① 本人が当事者か確認（reviewer_id = A）
② マッチング状態 = completed か確認
③ 決済成功か確認
④ 既にレビュー済みか確認（UNIQUE 制約）
⑤ 30日以内か確認
         ↓
バリデーション OK
↓
reviews テーブルに INSERT
↓
companies.rating_avg / review_count / rating_updated_at を UPDATE
↓
キャッシュをクリア（Redis の {company}:rating_avg を削除）
↓
201 Created で レスポンス返却
↓
フロントエンドで レビュー一覧を再フェッチ → 新しいスコアを表示

レビューの編集・削除

基本ポリシー

レビューは投稿後、編集・削除できない（不変性）。

理由

  • 会社の信頼スコアの操作防止
  • 評価履歴の完全性と監査対応
  • 荒らし行為の防止（投稿→削除→投稿 の繰り返し）

削除が必要な場合

  1. 不適切コンテンツ（誹謗中傷など）: Admin が レビュー ID で検索し、comment を "[削除済み - 不適切なコンテンツ]" に置き換え。rating は変更しない
  2. 誤投稿: Admin パネルで論理削除フラグを立てるが、統計には含まない

管理画面での操作

管理者（Admin）が利用可能な操作

  • レビュー一覧表示: 全レビューを時系列で表示
  • 不適切コンテンツ削除: comment フィールドを削除済みメッセージに置き換え
  • 偽装レビュー検出: 同じユーザーが短時間に複数投稿、など疑わしいパターンをフラグ
  • 会社のスコア再計算: 手動で rating_avg を再計算（データ修復用）

ユーザーが利用可能な操作

  • 投稿: 条件を満たしていれば投稿可能
  • 閲覧: 全てのレビューを閲覧可能
  • 編集・削除: 不可

実装例シナリオ

シナリオ: 運送会社A → 運送会社Bへの配送（マッチング）

  1. 配送会社A と 配送会社B がマッチング（match_id = 1000）
  2. 配送完了 → マッチング status = completed
  3. 配送会社A が配送会社B を評価: ★5 + コメント投稿
  4. レビュー INSERT: 
    INSERT INTO reviews (match_id, reviewer_id, reviewee_company_id, rating, comment)
VALUES (1000, 100, 200, 5, "プロフェッショナルな対応でした")
  5. companies テーブル更新: 
    UPDATE companies
SET rating_avg = 4.67, review_count = 15, rating_updated_at = NOW()
WHERE id = 200
  6. フロントエンド表示: 
    配送会社B
★★★★★ 4.67 (15件のレビュー)
最近の評価: "プロフェッショナルな対応でした"
