認証機能 (Authentication)

概要

delivery-api の認証機能は、ユーザー登録、ログイン、ログアウト、セッション管理を担当します。配送マッチングプラットフォームのすべてのユーザー（オーナー、ディスパッチャー、ドライバー、荷主、管理者）が安全に認証・認可される仕組みを提供します。

主な特徴:

  • JWT トークンベースの認証
  • HTTP-only Cookie によるトークン保存
  • リフレッシュトークンによるセッション管理
  • bcrypt によるパスワード暗号化
  • ユーザー列挙攻撃への対策
  • 3層アーキテクチャ (ハンドラー → サービス → リポジトリ)

APIエンドポイント

POST /api/v1/auth/register

新規ユーザーの登録

リクエスト

ボディ: JSON

{
  "email": "user@example.com",
  "password": "SecurePassword123",
  "name": "田中太郎",
  "role": "driver",
  "company_id": 1,
  "phone": "09012345678"
}

レスポンス

ステータス: 201 Created

{
  "success": true,
  "user": {
    "id": 5,
    "email": "user@example.com",
    "name": "田中太郎",
    "role": "driver",
    "company_id": 1,
    "phone": "09012345678",
    "is_active": true,
    "created_at": "2026-03-20T10:30:00Z"
  }
}

エラーレスポンス (400 Bad Request):

{
  "success": false,
  "error": "登録に失敗しました"
}

※ エラーメッセージは詳細を統一し、ユーザー列挙攻撃を防止

POST /api/v1/auth/login

ユーザーログイン

リクエスト

ボディ: JSON

{
  "email": "user@example.com",
  "password": "SecurePassword123"
}

レスポンス

ステータス: 200 OK

Set-Cookie ヘッダーで HTTP-only Cookie を設定

{
  "success": true,
  "user": {
    "id": 5,
    "email": "user@example.com",
    "name": "田中太郎",
    "role": "driver",
    "company_id": 1,
    "is_active": true
  },
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "refresh_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
}

エラーレスポンス (401 Unauthorized):

{
  "success": false,
  "error": "認証に失敗しました"
}

※ エラーメッセージは統一（メールアドレスが存在しないか、パスワードが間違っているか不明瞭にする）

POST /api/v1/auth/logout

ログアウト（クッキーをクリア）

リクエスト

認証: Bearer Token (JWT)

ボディ: 空

レスポンス

ステータス: 200 OK

Set-Cookie ヘッダーで auth_token と refresh_token を Max-Age=0 で削除

{
  "success": true,
  "message": "ログアウトしました"
}
POST /api/v1/auth/refresh

リクエスト

認証: refresh_token (HTTP-only Cookie)

ボディ: 空

レスポンス

ステータス: 200 OK

新しい access_token と refresh_token を発行（トークンローテーション）

{
  "success": true,
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "refresh_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."

エラーレスポンス (401 Unauthorized):

{
  "success": false,
  "error": "リフレッシュに失敗しました"

パスワードポリシー

パスワード仕様

  • 最小文字数: 6文字
  • 最大文字数: 72文字 (bcrypt の制限)
  • ハッシュ化: bcrypt (cost = 10)
  • ソルト: bcrypt が自動生成

パスワードは登録時に bcrypt でハッシュ化され、平文は保存されません。ログイン時には登録されたハッシュとの比較を行います。

クッキー仕様

クッキー名 内容 有効期限 フラグ 説明
auth_token JWT (access token) 15分 HttpOnly, Secure*, SameSite=Strict API リクエスト認証用の短期トークン
refresh_token JWT (refresh token) 7日 HttpOnly, Secure*, SameSite=Strict 新しい access_token を取得するためのトークン

* Secure フラグは本番環境では必須（HTTPS通信のみ）。開発環境では環境変数で制御可能。

セキュリティ考慮事項 重要

1. bcrypt ハッシング

  • パスワードはプレーンテキストで保存されない
  • cost パラメータ = 10（計算時間と安全性のバランス）
  • 72文字の制限に注意（登録時にバリデーション）

2. HTTP-only Cookie

  • JavaScript からアクセス不可（XSS 攻撃対策）
  • ブラウザが自動的にリクエストに含める（CSRF 対策と組み合わせ）
  • Secure フラグで HTTPS 通信のみに制限（本番環境）

3. エラーメッセージ統一化

  • ログイン失敗時は「認証に失敗しました」に統一
  • メールアドレスの存在有無を判定されない（ユーザー列挙攻撃対策）
  • ログは内部的に詳細を記録

4. リフレッシュトークンローテーション

  • 新しい access_token 取得時に refresh_token も更新
  • 古い refresh_token は使用不可に
  • 盗まれたトークンの悪用期間を短縮

5. トークン検証

  • 署名検証（HS256 またはRS256）
  • 有効期限の確認
  • ブラックリスト確認（ログアウト時）

6. CSRF 対策

  • SameSite=Strict でクロスオリジンリクエスト制限
  • POST リクエストでの状態変更

実装アーキテクチャ

3層アーキテクチャ

認証機能は以下の3層に分離します:

1. ハンドラー層 (Handler Layer)

  • 責務: HTTP リクエスト/レスポンス処理のみ
  • 処理内容:
    • リクエストボディのパース
    • 基本的なバリデーション（形式チェック）
    • クライアントからの認証情報抽出
    • レスポンスの構築と返却
    • HTTP ステータスコードの設定
  • 例: handlers/auth.go

2. サービス層 (Service Layer)

  • 責務: ビジネスロジック処理
  • 処理内容:
    • パスワード検証（bcrypt.CompareHashAndPassword）
    • JWT トークンの生成と署名
    • ユーザー登録ロジック（バリデーション、重複チェック）
    • トークンローテーション
    • セッション管理
  • 例: services/auth.go

3. リポジトリ層 (Repository Layer)

  • 責務: データベース操作
  • 処理内容:
    • ユーザーの検索（メールアドレスで）
    • ユーザーの作成
    • ユーザー情報の更新
    • トークンの保存
    • トークンの削除（ログアウト時）
  • 例: repositories/user.go

処理フロー例: ログイン

ハンドラー
  1. リクエストボディをパース → email, password を取得
  2. 形式バリデーション （email フォーマット、password 長）
  3. サービス層に処理依頼
         ↓
サービス
  4. リポジトリから email でユーザー検索
  5. bcrypt でパスワード検証
  6. JWT トークン生成
  7. トークンをリポジトリで保存（オプション）
  8. ユーザー情報とトークンを返却
         ↓
ハンドラー
  9. Set-Cookie ヘッダーを設定
  10. レスポンス JSON を作成
  11. 200 OK を返却
