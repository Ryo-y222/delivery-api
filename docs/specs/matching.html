マッチング仕様書

ドキュメント / 仕様書 / マッチング

概要

マッチング機能は、配送を依頼する荷主（Shipper）と、配送を実施する配送事業者（Company）を結びつけるコア機能です。 トリップの脚部（Trip Leg）に対して、空の状態で配送可能な容積がある場合、マッチング要求を作成し、 承認フロー、決済フロー、完了フローを経て配送を実現します。

データベーススキーマ

matches テーブル

カラム名 型 説明 制約
id UUID マッチング要求の一意識別子 PRIMARY KEY
trip_leg_id UUID 対象となるトリップの脚部ID NOT NULL, FOREIGN KEY
requester_id UUID マッチング要求者のユーザーID NOT NULL, FOREIGN KEY
requester_company_id UUID 要求者が属する企業ID（荷主の場合はNULL） NULLABLE, FOREIGN KEY
cargo_weight DECIMAL(10,2) 荷物の重量（kg） NOT NULL
cargo_description TEXT 荷物の説明 NOT NULL
message TEXT 要求者からのメッセージ NULLABLE
status ENUM マッチングの状態 pending | approved | payment_pending | completed | rejected | cancelled
reject_reason TEXT 却下理由（拒否時のみ） NULLABLE
request_type ENUM 要求のタイプ shipper_to_company | company_to_company
created_at TIMESTAMP 作成日時 NOT NULL
updated_at TIMESTAMP 更新日時 NOT NULL

要求タイプ

  • shipper_to_company: 荷主が配送事業者に配送を依頼
  • company_to_company: 企業が別の企業に配送を依頼

ステータス状態遷移

状態定義

  • pending: 新規作成、承認待ち状態
  • approved: 配送事業者が承認した状態。決済前
  • payment_pending: 決済処理中の状態
  • completed: 決済完了、配送実施完了
  • rejected: 配送事業者が却下した状態
  • cancelled: キャンセルされた状態

状態遷移図

┌─────────┐
│ pending │
└────┬────┘
     │
     ├─→ approved ─→ payment_pending ─→ completed
     │
     ├─→ rejected
     │
     └─→ cancelled (any state)

【重要】
- approved から直接 completed への遷移は禁止
- 必ず payment_pending を経由する必要がある
- rejected および cancelled は他の状態から到達可能

遷移ルール

現在の状態 遷移先 条件・イベント
pending approved 配送事業者が承認
pending rejected 配送事業者が却下
pending cancelled 要求者がキャンセル
approved payment_pending 決済フロー開始
approved cancelled いずれかが合意してキャンセル
payment_pending completed 決済完了（Stripe webhook）
payment_pending cancelled 決済キャンセル

マッチング対象の検索ロジック

検索条件

マッチング対象となるTrip Legは以下の条件を満たす必要があります：

WHERE
    cargo_status = 'empty_seeking'
    AND visibility IN ('public', 'company_only')

フィルター API

マッチング要求時に以下のフィルターが使用可能です：

  • area: 配送地域（緯度経度範囲など）
  • date: 配送予定日
  • vehicle_type: 車両タイプ（トラック、バン等）
  • available_weight: 利用可能な容積・重量

承認時の自動処理

マッチングが承認される際、以下が自動実行されます：

  1. ステータスを approved に変更
  2. 要求者と承認者間のチャットルームを自動作成
  3. 決済フロー用のPaymentIntentを準備

API エンドポイント

マッチング要求を作成

POST /api/v1/matches

新しいマッチング要求を作成します。

リクエストボディ

{
  "trip_leg_id": "uuid",
  "requester_id": "uuid",
  "requester_company_id": "uuid or null",
  "cargo_weight": 100.5,
  "cargo_description": "電子機器 1箱",
  "message": "お忙しいところ恐れ入りますが、この荷物をお送りいただけますでしょうか",
  "request_type": "shipper_to_company"
}

レスポンス (201 Created)

{
  "id": "uuid",
  "trip_leg_id": "uuid",
  "requester_id": "uuid",
  "status": "pending",
  "created_at": "2026-03-20T10:30:00Z",
  "updated_at": "2026-03-20T10:30:00Z"
}

マッチング要求を取得

GET /api/v1/matches/:id

指定IDのマッチング要求の詳細情報を取得します。

レスポンス (200 OK)

  "updated_at": "2026-03-20T10:30:00Z"

マッチング要求を承認

PUT /api/v1/matches/:id/approve

マッチング要求を承認します。ステータスを pending → approved に変更し、チャットルームを自動作成します。

リクエストボディ

{
  "approver_id": "uuid"
}

レスポンス (200 OK)

{
  "id": "uuid",
  "status": "approved",
  "chat_room_id": "uuid",
  "updated_at": "2026-03-20T10:35:00Z"
}

マッチング要求を却下

PUT /api/v1/matches/:id/reject

マッチング要求を却下します。

リクエストボディ

{
  "reason": "容積が不足しています"
}

レスポンス (200 OK)

{
  "id": "uuid",
  "status": "rejected",
  "reject_reason": "容積が不足しています",
  "updated_at": "2026-03-20T10:35:00Z"
}

ユーザーのマッチング要求一覧を取得

GET /api/v1/matches?requester_id=:id

指定されたユーザーによるマッチング要求の一覧を取得します。

クエリパラメータ

  • requester_id: フィルター対象のユーザーID
  • status (optional): ステータスでフィルター
  • limit (optional): 取得件数の上限（デフォルト: 20）
  • offset (optional): オフセット（デフォルト: 0）

レスポンス (200 OK)

{
  "items": [
    {
      "id": "uuid",
      "trip_leg_id": "uuid",
      "status": "pending",
      "cargo_description": "...",
      "created_at": "2026-03-20T10:30:00Z"
    }
  ],
  "total": 5,
  "limit": 20,
  "offset": 0
}

流れの詳細

  1. 検索: ユーザーがフィルター条件を指定して空きトリップを検索
  2. 要求作成: マッチング要求を作成（ステータス: pending）
  3. 確認: 配送事業者がマッチング要求を確認して承認または却下
  4. 決済開始: 承認時に决済フロー用のPaymentIntentを作成
  5. 決済実行: ユーザーがクレジットカード情報をStripeに送信
  6. 完了: Stripe webhookで決済完了を確認し、ステータスを completed に変更

重要な注意点

  • payment_pendingの必須性: approved ステータスから直接 completed に遷移することは禁止されています。 必ず payment_pending を経由して決済フローを完結させる必要があります。
  • RequesterCompanyIDの扱い: 個人の荷主（Shipper）の場合、requester_company_id は NULL となります。 企業間マッチングの場合のみ値が入ります。
  • チャットルームの自動作成: マッチングが承認される際に、要求者と承認者のチャットルームが自動作成されます。 その後の配送に関するやり取りはこのチャットルームで行われます。
  • キャンセル可能性: どのステータスからでも cancelled への遷移は可能ですが、 payment_pending 以降のキャンセルには払い戻し処理が発生する場合があります。
