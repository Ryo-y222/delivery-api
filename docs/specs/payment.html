決済仕様書

ドキュメント / 仕様書 / 決済

概要

決済機能は、マッチングが承認された後、荷主から配送事業者への支払いを処理するコア機能です。 Stripe PaymentIntent APIを使用して安全なクレジットカード決済を実現します。 決済完了後、自動的にマッチングのステータスを completed に更新し、配送フローを完結させます。

データベーススキーマ

payments テーブル

カラム名 型 説明 制約
id UUID 決済レコードの一意識別子 PRIMARY KEY
match_id UUID 対象となるマッチング要求のID NOT NULL, FOREIGN KEY
payer_id UUID 支払い者（荷主）のユーザーID NOT NULL, FOREIGN KEY
amount INTEGER 決済金額（最小単位：円） NOT NULL, CHECK (amount > 0)
currency VARCHAR(3) 通貨コード（ISO 4217） NOT NULL, DEFAULT 'jpy'
stripe_payment_id VARCHAR(255) Stripe PaymentIntent ID UNIQUE, NOT NULL
status ENUM 決済ステータス pending | succeeded | failed | refunded
paid_at TIMESTAMP 決済完了日時 NULLABLE
created_at TIMESTAMP レコード作成日時 NOT NULL
updated_at TIMESTAMP レコード更新日時 NOT NULL

データ型の重要な注意

amount は INTEGER であることの理由

決済金額は 浮動小数点型（FLOAT/DECIMAL）ではなく INTEGER 型 を使用します。 理由は以下の通りです：

  • 浮動小数点の丸め誤差: 浮動小数点演算では 0.1 + 0.2 ≠ 0.3 という問題が発生します。 決済システムでの誤差は重大な問題となるため、整数値で金額を管理します。
  • Stripeとの互換性: Stripe APIも金額を最小単位（日本円の場合は1円）の整数として扱います。 データベースのデータ型を Stripe の形式に合わせることで、 変換ロジックの簡潔性と正確性を確保します。

amount = 10000 は 10,000円を表します。

paid_at は TIMESTAMP NULLABLE である理由

paid_at は *time.Time (Goの NULL許可ポインタ型) としてNULLABLEです。 理由は、決済が完了していない状態（pending / failed）では、実際の支払い日時が存在しないためです。

決済ステータス

ステータス定義

  • pending: PaymentIntentが作成され、フロントエンドがカード情報をStripeに送信するのを待っている状態
  • succeeded: 決済が正常に完了した状態。Stripe webhookで確認される
  • failed: 決済が失敗した状態（カード拒否、有効期限切れ等）
  • refunded: 一度完了した決済が払い戻された状態

ステータス遷移図

┌─────────┐
│ pending │
└────┬────┘
     │
     ├─→ succeeded ─→ refunded
     │
     └─→ failed

【フロー】
1. PaymentIntentを作成 → pending
2. フロントエンドがカード情報をStripeに送信
3. Stripe決済成功 ↔ webhook通知 → succeeded
4. Stripe決済失敗 → webhook通知 → failed
5. 払い戻し処理 → refunded

Stripe PaymentIntent フロー

PaymentIntentの流れ

  1. Backend: PaymentIntentを作成
    マッチング承認時に、バックエンドが Stripe PaymentIntent API を呼び出して PaymentIntentを作成します。金額、通貨、メタデータが設定されます。
  2. Frontend: Stripe Card Form を表示
    バックエンドから返されたclient_secretを使用して、 フロントエンドがStripe Elements / Stripe.jsを用いたカード入力フォームを表示します。
  3. Frontend: カード情報をStripeに送信
    ユーザーがカード情報を入力し、「支払う」ボタンをクリック。 フロントエンドはこの情報を直接Stripeサーバーに送信します（バックエンドを経由しない）。
  4. Stripe: 決済処理を実行
    Stripeがカード決済を処理し、成功または失敗の結果を確定します。
  5. Stripe: Webhookで通知
    決済結果が確定したら、Stripeはバックエンドのwebhookエンドポイントに payment_intent.succeeded または payment_intent.payment_failed イベントを送信します。
  6. Backend: DBを更新し、マッチングを完了
    webhookで成功を確認したら、paymentsテーブルのステータスを succeeded に更新。 同時に、対応するマッチングのステータスを completed に更新します。

Webhook ハンドリング

Webhookエンドポイント

POST /api/v1/webhooks/stripe

対象イベント

イベント名 説明 処理内容
payment_intent.succeeded 決済が成功した 1. payments テーブルの status を succeeded に更新
2. paid_at に現在時刻を記録
3. 対応するマッチングの status を completed に更新
4. メール通知を配信
payment_intent.payment_failed 決済が失敗した 1. payments テーブルの status を failed に更新
2. 失敗原因をログに記録
3. ユーザーに失敗通知メールを送信
4. マッチングのステータスは payment_pending のまま保持
（ユーザーが再度決済を試みることが可能）

Webhook署名検証

セキュリティのため、Stripeからのwebhook通知は 必ず署名を検証 する必要があります。 

【署名検証の手順】
1. Webhook リクエストの body（生のバイナリ）を取得
2. リクエストヘッダーから署名情報を抽出
   - Stripe-Signature ヘッダー
   - フォーマット: t=タイムスタンプ,v1=署名値,...
3. バックエンドで同じアルゴリズムを用いて署名を再計算
4. リクエストの署名と再計算署名を比較
5. 一致すればStripeからの正規なリクエストと確認

【理由】
未検証のwebhookを受け入れると、攻撃者が偽装の決済完了通知を送信でき、
決済なしでマッチングが完了してしまう危険があります。

署名検証に使用するシークレット

Stripe ダッシュボードから取得した Webhook Signing Secret を使用します。 このシークレットは環境変数 STRIPE_WEBHOOK_SECRET として安全に管理してください。

API エンドポイント

PaymentIntentを作成

POST /api/v1/payments/create-intent

マッチング承認後、支払い者がPaymentIntentを作成してフロントエンドで決済フォームを表示するために必要です。

リクエストボディ

{
  "match_id": "uuid",
  "payer_id": "uuid",
  "amount": 10000,
  "currency": "jpy"
}

レスポンス (200 OK)

{
  "payment_id": "uuid",
  "client_secret": "pi_xxxx_secret_yyyy",
  "stripe_payment_id": "pi_xxxx",
  "amount": 10000,
  "currency": "jpy",
  "status": "pending"
}

エラーレスポンス (400 Bad Request)

{
  "error": "Invalid match status",
  "details": "マッチングが approved ステータスではありません"
}

Stripe Webhook 受信エンドポイント

POST /api/v1/webhooks/stripe

Stripeから決済結果の通知を受け取り、データベースを更新します。 このエンドポイントは認証を必要としません（代わりに署名で検証）。

リクエストヘッダー

Content-Type: application/json
Stripe-Signature: t=1614556800,v1=7c6f5eff7f6b8c9d0e1f2a3b4c5d6e7f

リクエストボディ例

{
  "type": "payment_intent.succeeded",
  "data": {
    "object": {
      "id": "pi_xxxx",
      "amount": 10000,
      "currency": "jpy",
      "status": "succeeded",
      "metadata": {
        "match_id": "uuid",
        "payer_id": "uuid"
      }
    }
  }
}

レスポンス (200 OK)

{
  "received": true
}

エラーレスポンス (400 Bad Request / 401 Unauthorized)

{
  "error": "Invalid signature"
}

フロー説明

  1. マッチングが承認されたら、バックエンドがPaymentIntentを作成
  2. フロントエンドが client_secret を使用してStripe Card Formを初期化
  3. ユーザーがカード情報を入力し、フロントエンドがStripeに直接送信
  4. Stripeが決済を処理し、成功/失敗を確定
  5. Stripeがバックエンドにwebhookで通知
  6. バックエンドがwebhookの署名を検証してから、DBを更新
  7. 成功時：マッチングを completed に更新、ユーザーに完了通知
  8. 失敗時：ステータスは payment_pending のまま、ユーザーは再試行可能

セキュリティ考慮事項

1. PCI DSS コンプライアンス

  • カード情報は絶対にバックエンドで扱わない: Stripe.jsを使用してカード情報をフロントエンドから直接Stripeに送信します。 バックエンド経由でカード情報が通過しないため、PCI DSSの適用範囲外となります。
  • ログにも決済情報の詳細は記録しません。stripe_payment_idのみ記録。

2. Webhook署名検証

  • すべてのStripeからのwebhookは 必ず署名を検証 してください。
  • 攻撃者が偽装webhookを送信することで、決済なしでマッチングが完了する攻撃を防止します。
  • 検証失敗時は通知を拒否し、エラーログに記録してください。

3. IdempotencyとDuplicate Prevention

  • 同じPaymentIntentの複数回の webhook受信に備えて、 DB更新時にidempotent（冪等）な処理を実装してください。
  • 例: ステータスが既に succeeded なら、再度更新しない

4. HTTPS通信

  • すべてのAPI通信はHTTPS経由で行い、中間者攻撃を防止してください。
  • webhookエンドポイントはHTTPではなくHTTPSで公開してください。

5. 環境変数管理

  • STRIPE_SECRET_KEY と STRIPE_WEBHOOK_SECRET は 本番環境で環境変数として安全に管理してください。
  • ソースコードに直接記述しないでください。

重要な注意点

  • 金額は整数で管理: 浮動小数点の丸め誤差を避けるため、金額は常に整数（円単位）で扱います。 10,000円は amount = 10000 です。
  • paid_at はNULL可能: 決済完了まで paid_at は NULL です。 succeeded ステータス確認後に、webhookで更新されます。
  • payment_pending は必須: マッチングが approved → payment_pending → completed の流れを厳密に遵守してください。
  • Webhook署名検証は必須: webhookの署名検証なしに処理を進めると、セキュリティが大きく損なわれます。
  • インターネット通信の遅延を考慮: webhookが届く前にユーザーが画面をリロードしても、最終的には webhookで状態が確定するため、フロントエンドはポーリングまたは Real-time通知で最新状態を取得してください。
