配車区間 (Trip Leg) 機能仕様書

ドキュメント 配車計画
ドキュメント / 配車区間

概要

配車区間 (Trip Leg) は、配送プラットフォームのマッチングエンジンの中核

Trip Legの役割

  • 運行単位の最小粒度: 地理的・時間的な単一セグメント
  • マッチング対象の定義: cargo_status と visibility の組み合わせで、誰がこの輸送能力を利用できるかを決定
  • 経路情報の保持: Google Routes API による詳細なステップ情報
  • 位置追跡対象: 各TripLegの進捗をリアルタイムに監視

CREATE INDEX idx_trip_legs_matching ON trip_legs(visibility, cargo_status, status);

CREATE INDEX idx_trip_legs_dispatch_plan ON trip_legs(dispatch_plan_id, leg_order);

荷物状態 (Cargo Status)

TripLegのcargo_statusは、この区間における車両の荷物積載状態を表し、マッチング可否を判定する最重要フィールドです。

ステータス 説明 マッチング対象 用途例
loaded 既に荷物を積んでいる状態 ❌ 対象外 配送計画の往路区間。この荷物は既に確定されているため、新たなマッチングは発生しない
empty_seeking 空車で次の荷物を探索中 ✅ 対象 復路区間。この区間中、他の荷主からの積荷リクエストと照合される
empty_private 空車だが非公開（回送等） ❌ 対象外 営業所への回送。マッチング対象にしない重要な私有運行

マッチング条件

マッチングエンジンが照合するTripLegの条件:

WHERE cargo_status = 'empty_seeking'
  AND visibility IN ('public', 'company_only')
  AND status = 'scheduled'
  AND departure_at > NOW()
  AND available_weight >= required_weight

公開範囲 (Visibility)

visibilityフィールドは、このTripLegを誰がマッチング対象として見られるかを制御します。

レベル 説明 見える対象 マッチング対象
private 非公開（自社運行のみ） 当該企業の管理者のみ ❌ 対象外
company_only 同一企業内のみ公開 同じcompany_idを持つ他の利用者 ✅ 企業内マッチング用
public 完全公開 全プラットフォーム利用者 ✅ オープンマッチング用

区間型 (Leg Type)

型 説明 cargo_status マッチング
outbound 往路（配送地への移動） 通常はloaded ❌ 対象外
return 復路（配送地からの移動） empty_seekingまたはempty_private status次第で可

経路データ

route_polyline (TEXT)

Google Maps Encoded Polyline形式の経路情報。フロントエンドで地図上に描画します。

例: "_p~iF~ps|U_ulLnnqC_mqNvxq`@"

route_duration_sec (INT)

Google Routes APIが返す推定所要時間（秒）。到着予測時刻の計算に使用。

route_steps_json (MEDIUMTEXT)

Google Routes APIが返す詳細ステップをJSON形式で保存

理由: Google Routes APIの各ステップ情報（「〇〇方向に進む」「右折」など）は量が多く、TEXTでは不足するためMEDIUMTEXTを使用。

{
  "steps": [
    {
      "startLocation": { "latitude": 35.4437, "longitude": 139.6380 },
      "endLocation": { "latitude": 35.4450, "longitude": 139.6390 },
      "polyline": { "encodedPolyline": "..." },
      "distanceMeters": 150,
      "duration": "12s",
      "staticDuration": "12s",
      "navigationInstruction": "Head south on Route 1",
      "maneuver": "STRAIGHT"
    },
    ...
  ]
}

インデックス戦略の詳細

複合インデックス idx_trip_legs_matching

マッチング検索のパフォーマンスを最適化するために、3つのカラムを結合したインデックスを使用します。

CREATE INDEX idx_trip_legs_matching
ON trip_legs(visibility, cargo_status, status);

このインデックスにより、以下のクエリが高速化される:
SELECT * FROM trip_legs
WHERE visibility IN ('public', 'company_only')
  AND cargo_status = 'empty_seeking'
  AND status = 'scheduled';

インデックスの列の順序は重要です。最初にvisibilityを指定することで、パブリック・コンパニー限定の迅速なフィルタリングが可能になります。

設計ノート

route_steps_json は MEDIUMTEXT

Google Routes APIが返す詳細ステップ情報は大量のJSON構造を含むため、通常のTEXT型では不足する可能性があります。MEDIUMTEXTを使用することで、最大16MBまでの経路情報を保存できます。

cargo_status の3値設計

loaded / empty_seeking / empty_private の3つの状態により、マッチング対象性を明確に制御できます。これにより、営業所への回送（empty_private）とマッチング可能な空車（empty_seeking）を区別できます。

visibility と cargo_status の相互関係

マッチング可能な条件は両方のフィールドに依存します。visibility だけでなく cargo_status も 'empty_seeking' である必要があり、その他の値では決してマッチング対象になりません。

delay_minutes の使用例

実際の運行中に予定との差が生じた場合、delay_minutesを更新します。これにより到着予測の精度が向上し、次のマッチングの判定に活用できます。
